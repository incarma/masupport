# django_ma 성능·보안 등급 분류 지침서

## 1. 목적
본 문서는 django_ma 프로젝트에서 발견되는 보안/성능/구조 이슈를
일관된 기준으로 분류하기 위한 등급 체계이다.

이 문서는 별도의 설명 없이도 동일 기준으로 판단 가능하도록 설계된다.

---

## 2. 등급 체계 정의

### 🔴 최상 (Critical)
- 보안 사고 또는 데이터 유출 가능
- 권한 검증 실패
- 인증/세션/파일 접근 취약점

#### 포함 항목
- 파일 다운로드 직접 접근 구조
- 업로드 결과 파일 권한 미검증
- X-Forwarded-For 신뢰 검증 없음
- CSP unsafe-inline / unsafe-eval 허용

---

### 🟠 상 (High)
- 서비스 장애 가능성
- DB 과부하 또는 대량 처리 비효율

#### 포함 항목
- RequestLog 과도한 저장
- Excel row-by-row 처리
- 검색 API full scan (icontains 남용)

---

### 🟡 중상 (Moderate-High)
- 트래픽 증가 시 성능 저하
- 장기 운영 시 장애 발생 가능

#### 포함 항목
- Audit meta 크기 무제한
- Celery idempotency 미보장
- temp 파일 정리 없음
- 비효율적인 index 구조

---

### 🟢 중 (Moderate)
- 누적 시 성능 저하 발생
- 일부 기능 비효율

#### 포함 항목
- signals 중복 DB 조회
- SubAdminTemp sync 비효율
- DataTables 전체 재렌더링

---

### 🔵 중하 (Low-Mid)
- 유지보수 리스크 증가
- 코드 구조 비일관성

#### 포함 항목
- admin.py 비대화
- fetch JSON 처리 중복
- submit lock 로직 분산

---

### ⚪ 하 (Low)
- 구조 개선 및 최적화 영역
- 즉각적인 영향 없음

#### 포함 항목
- Docker 이미지 최적화
- vendor unused 제거
- 설정 문서화

---

## 3. 적용 원칙

1. 모든 이슈는 반드시 위 등급 중 하나로 분류한다.
2. 등급 판단 기준은 "영향 범위 + 장애 가능성"을 따른다.
3. 보안 관련 이슈는 항상 최우선으로 평가한다.
4. 동일 이슈라도 운영 환경 기준으로 판단한다.

---

## 4. 우선순위 적용 순서

1. 최상 (보안)
2. 상 (DB/업로드)
3. 중상 (로그/비동기/파일)
4. 중 (쿼리/프론트)
5. 중하 (구조)
6. 하 (정리/최적화)

---

## 5. 핵심 원칙 요약

- 보안 > 성능 > 구조 순으로 판단
- 파일 접근은 반드시 view + 권한검증
- DB write는 최소화
- 공용 로직은 SSOT로 유지
- 프론트는 dataset 기반 Boot 패턴 유지
